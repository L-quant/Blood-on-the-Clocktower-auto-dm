# 暗流涌动 — 夜晚阶段后端逻辑需求文档 (AI DM 重构版)

> 基于官方夜晚顺序表（clocktower-wiki.gstonegames.com）及 14 阶段重构计划最新结论。
> 本游戏仅支持 AI 自动主持（无人类说书人），夜晚逻辑采用“收集-结算-分发”三层架构。

---

## 1. 核心架构逻辑：三层架构

为了解决现有引擎中“即时结算”引发的时序与状态不一致问题，夜晚逻辑必须分为解耦的三层：

1. **行动收集层 (Collection Layer)**：
   - 按 Night Order 依次下发 `night.action.prompt`。
   - 等待 `ability.use`，将目标记录为 `PendingAction`，发送 `night.action.completed`。
   - **不进行任何实质性状态改变，不产生玩家死亡，不生成信息**。
2. **结算层 (Resolution Layer)**：
   - 所有人行动收集完毕后（即全部转入无提示等待状态），统一触发 `resolveNight()`。
   - **按官方具体结算顺序遍历执行**，如：投毒者先行结算 -> 分配中毒标记；恶魔结算 -> 判定死者。
   - **重要规则 - 死亡回滚**：如果角色当晚死亡（如被投毒者当晚被恶魔击杀或因特殊事件死亡），其当晚主动能力失效。例如投毒者今晚对A下毒，但投毒者今晚死了，则A的中毒状态需要回滚失效！
   - 特殊拦截：在此阶段如果触发了因死亡唤醒的角色（如守鸦人），会直接在此刻临时挂起，要求守鸦人输入目标，获取目标后再继续后续结算。
3. **信息分发层 (Distribution Layer)**：
   - 依赖结算完成后的**最终快照状态**（包括最终存活状态、最终中毒状态），统一由 `distributeNightInfo()` 分发信息。
   - 确保诸如“厨师”、“共情者”获取的是当晚恶魔杀人后的最新状态，且不会因为说书人即时处理而泄漏时序。
   - 生成如 `night.info`（发给信息位或提醒位）或 `team.recognition`（第一夜邪恶阵营互认）。

## 2. 玩家状态模型

```go
type PlayerState struct {
    ID                      string
    Role                    string
    Alignment               string   // "good" | "evil"
    IsAlive                 bool
    IsExecutedToday         bool     // 今天白天被处决

    // 跨夜持久化
    Poisoned                bool     // 投毒者标记，每夜轮换
    MonkProtected           bool     // 僧侣保护，每夜天亮重置
    ButlerMasterID          string   // 管家主人ID，每夜可换
    SpyApparentRole         string   // 间谍对信息角色显示的假身份：引擎Setup时从不在场的善良角色中随机指定
}

type GameState struct {
    Players                    []PlayerState
    FortuneTellerRedHerring    string   // Setup固定，占卜师"干扰项"玩家ID
    NightNumber                int
    ScarlettWomanTriggered     bool     // 红唇女郎是否已继承，防重复触发
}

// 收集层的动作记录意图
type NightActionIntent struct {
    PlayerID string
    Role     string
    TargetID []string
}
```

### 字段层级与 AI 决策说明

| 字段名/机制 | 级别 | AI 处理方式 |
|--------|------|----------|
| `SpyApparentRole` | 间谍系统 | **随机自动分配**：由引擎在 Setup 时，自动从局外未上场的善良角色中随机分配，不再由 AI Agent 推理，确保即时可用。 |
| `MayorTransfer` | 镇长转移 | **AI随机决策**：当镇长受击且满足转移条件时，目前引擎自动从存活角色中（或交由AI Agent）随机选择目标转移。 |
| `FortuneTellerRedHerring` | 干扰项 | 由引擎在 Setup 阶段随机从存活的善良村民（非恶魔）中指定，整局不变更。 |

## 3. 分层执行流程规范

### A. 第一夜（Setup 后首夜）

| 角色/阶段 | 层级 | 处理方式 |
|------|------|---------|
| 邪恶互认 | **信息分发** | 推送 `team.recognition` 事件，包含爪牙互认与恶魔认爪牙（不在场善良角色自动随机抽取并分发）。仅 ≥7 人局生效。 |
| 投毒者 | **收集 -> 结算** | 收集目标 ID -> 结算时为目标写入 `Poisoned = true`。 |
| 间谍 | **信息分发** | 收集层无动作。结算后由系统自动将包含所有玩家最新状态的 `GrimoireSnapshot` 推送给间谍。 |
| 所有信息位 | **收集 -> 分发** | 洗衣妇/图书管理员/调查员/厨师/共情者/占卜师 在收集层无须提交或仅提交目标；结算后系统根据是否 `Poisoned` 来计算并推送 `night.info` 假信息（若中毒）。 |
| 管家 | **收集 -> 结算** | 收集主人 ID -> 写入 `ButlerMasterID`。 |

### B. 常规夜晚（循环）

| 角色/阶段 | 层级 | 处理方式 |
|------|------|---------|
| 投毒者 | **收集 -> 结算** | 先重置旧目标，收集新目标，结算写入 `Poisoned`。**若投毒者当晚被杀，该毒效将被回滚失效**。 |
| 僧侣 | **收集 -> 结算** | 收集目标，写入 `MonkProtected=true`。若僧侣自身已中毒，仍收集并写入，但结算优先级中无效化。 |
| 小恶魔 | **收集 -> 结算** | 收集击杀目标 -> 在结算层执行 `resolveDemonKill`。包含击杀自己触发红唇女郎继承机制。 |
| 守鸦人/送葬者 | **特殊收集 -> 分发** | 这两个角色不出现在常规收集队列。只有在**结算层**判定为触发条件时，才会发送 `night.continue` 提示，收集目标后再予以反馈（分发真假角色信息）。 |
| 其他信息位 | **分发** | 共情者/占卜师 等收集后，按结算后的最终格局查数据并推送。 |

## 4. 死亡结算函数 `resolveDemonKill` (统一结算层)

```go
func resolveDemonKillInner(targetID string, state *GameState) {
    target := state.Players[targetID]

    // 优先级1：士兵免疫（被动，中毒时失效）
    if target.Role == "士兵" && !target.Poisoned {
        return // 不死
    }

    // 优先级2：僧侣保护（僧侣自身中毒时保护无效）
    monk := findPlayerByRole(state, "僧侣")
    monkHealthy := monk != nil && !monk.Poisoned
    if target.MonkProtected && monkHealthy {
        return // 不死
    }

    // 优先级3：镇长转移（引擎或AI Agent自动裁量随机转移目标）
    if target.Role == "镇长" {
        otherID := AutoSelectMayorTransferTarget(state) // AI自动化处理，随机存活玩家
        state.Players[otherID].IsAlive = false
        checkPostDeath(otherID, state)
        return // 镇长不死
    }

    // 默认：目标死亡
    target.IsAlive = false
    checkPostDeath(targetID, state)
}

func checkPostDeath(playerID string, state *GameState) {
    player := state.Players[playerID]

    // 小恶魔自杀触发红唇女郎继承
    if playerID == state.DemonID {
        resolveScarlettWomanInherit(state)
    }
    
    // 如果死者是投毒者，其当晚下毒效果应当回滚
    if player.Role == "投毒者" {
        rollbackPoisonEffect(state)
    }
}
```

## 5. 红唇女郎继承 `resolveScarlettWomanInherit`

**触发时机**：小恶魔死亡（被处决或在**结算层**主动选择自己为目标时判定）。

```go
func resolveScarlettWomanInherit(state *GameState) {
    if state.ScarlettWomanTriggered {
        return // 防重复触发
    }

    // 条件：存活玩家数（不含旅行者）>= 5
    aliveCount := countAliveNonTravellers(state)
    if aliveCount < 5 {
        return // 不触发，善良方直接胜利
    }

    sw := findAlivePlayerByRole(state, "红唇女郎")
    if sw == nil {
        return // 红唇女郎不在场或已死
    }

    // 继承逻辑
    sw.Role = "小恶魔"
    state.DemonID = sw.ID
    state.ScarlettWomanTriggered = true

    // 此信息最终会在分发层推送给红唇女郎
    queueNightInfo(sw.ID, "你现在是小恶魔", true)
}
```

## 6. 间谍魔典快照 `GrimoireSnapshot`

间谍每夜由**分发层**自动推送完整魔典。必须在死亡结算与毒药结算完成后进行，确保反映事实。

```go
type GrimoireSnapshot struct {
    Players []PlayerSnapshot
}

type PlayerSnapshot struct {
    ID        string
    SeatIndex int
    Role      string    // 真实角色
    Alignment string    // "good" | "evil"
    IsAlive   bool
    Poisoned  bool
    Reminders []string  // 当前所有 reminder token，如"中毒"、"被保护"
}
```

## 7. 信息生成与间谍干扰规则

所有信息类角色（在分发层处理）不能直接读取真实 `Alignment` 或 `Role`，必须经过安全封装：

```go
// 间谍对所有查验阵营的作用返回 "good" (善良)
func getApparentAlignment(playerID string, state *GameState) string {
    if state.Players[playerID].Role == "间谍" {
        return "good"
    }
    return state.Players[playerID].Alignment
}

// 间谍返回其在Setup时分配的随机善良角色（对调查员、洗衣妇等生效）
func getApparentRole(playerID string, state *GameState) string {
    p := state.Players[playerID]
    if p.Role == "间谍" && p.SpyApparentRole != "" {
        return p.SpyApparentRole
    }
    return p.Role
}
```

**影响范围：**
- **厨师/共情者**：阵营判定调用 `getApparentAlignment`，间谍混同于好人。
- **调查员/洗衣妇**：角色判断调 `getApparentRole`，间谍甚至能混淆调查员判定。

## 8. 中毒后的虚假信息生成规则

如果信息角色 `Poisoned=true`，引擎的**分发层**自动为其生成合理但“大概率错误”的虚假信息。由于本应用是 AI 说书人全自动处理，采用以下策略防范人类玩家推断出假结论：
- 数值型（厨师/共情者）：对真实数值进行随机偏移（+1 或 -1，防止越界或产生不合理结果）。
- 布尔型（占卜师）：以 100% 概率翻转其真实验血结果。
- 文本型（各类信息村民获取指定身份）：随机生成场外的对应符合要求的非目前实际状态的结果。

## 9. 前后端交互事件表 (事件溯源对应)

- **收集阶段（后端 -> 前端）**
  - `night.action.prompt`：下发输入表单（不再包含 result）
- **玩家意图（前端 -> 后端）**
  - `ability.use`：传入目标群体 `[]string`
- **确认进入就绪（后端 -> 前端）**
  - `night.action.completed`：反馈前端变为「sleeping」视图（不包含实质结果）
- **特殊中途挂起请求（后端 -> 前端）**
  - `night.continue`：部分角色因为前序角色触发而突然唤醒（守鸦人）
- **结算与分发阶段（后端 -> 前端）**
  - `team.recognition`：向对应人分发开局互认信息。
  - `night.info`：向所有带有结果的角色推送结算信息。前端收到后切为 `result` 视图弹窗。

## 10. 边界情况汇总

| 情况 | 处理方式 |
|------|---------|
| 僧侣保护了酒鬼 | 酒鬼受保护逻辑正常生效；酒鬼自身的"不是真实角色"属性不影响死亡结算 |
| 僧侣自身中毒 | 僧侣仍正常执行选人操作，但 `MonkProtected` 标记在结算时被引擎无视 |
| 士兵中毒 | 士兵免疫失效，恶魔可正常击杀 |
| 投毒者当晚被杀 | **其当晚的投毒效果会被回滚失效**，信息角色拿到真信息 |
| 占卜师选中 Red Her. | 该玩家被当作恶魔处理，返回"是"；玩家死亡依旧生效 |
| 红唇处于死态或人数 < 5 | 小恶魔自杀不触发继承，若无恶魔则善良方立刻判断胜利 |
| 守鸦人被恶魔杀死 | **拦截结算**，发送 `night.continue` 索要守鸦人目标，返回结果后再往下结算 |
| 送葬者自己被处决 | 送葬者死亡不唤醒，无法获得自己的被处决信息 |
| 间谍被共情者探查 | `getApparentAlignment` 返回 "good"，间谍算善良玩家 |
