# 修复游戏流程，确保“可完整跑完一局”且不返工

## Context
实际联调时，游戏无法稳定走完完整循环（大厅 → 首夜 → 白天 → 提名/辩护/投票 → 夜晚 → 循环 → 结束）。  
除已发现问题外，还存在会导致“修一处又返工”的隐性漏洞，需要先补齐。

### 已确认阻塞点
1. 处决恶魔后，`handleVote` 路径下胜利条件检测使用旧状态，可能不触发 `game.ended`。  
2. 前端无“结束辩护”操作，`defense` 子阶段可永久卡死。  
3. 白天无“进入夜晚”操作，白天可永久卡死。  
4. `handleCloseVote` 与 `resolveNomination` 事件/字段不一致（`player.executed` vs `player.died`、`yes_votes` vs `votes_for`）。  
5. 投票阈值在两处计算不一致（`(alive/2)+1` vs `(alive+1)/2`）。  
6. `nomination.created` 时前端 `requiredMajority=0`，投票进度失效。  
7. `VoteOverlay` 依赖 `isMyTurn` 才显示投票按钮，但当前代码没有任何路径会把它置 `true`，导致玩家可能无法投票。  
8. “进入夜晚”按钮若依赖 `!isVoteActive`，在 `nomination.resolved` 后仍可能不出现（状态未及时收敛）。  
9. 计划中的“辩护超时自动结束”会被后端权限拦截：`autodm` 目前不能调用 `end_defense`。  
10. `autodm` 代理提名时，提名者 actor 与状态更新来源不一致，可能污染 `HasNominated/WasNominated` 与相关事件语义。

### 本次目标（验收标准）
1. 5 人局可手动稳定走完一局，不需人工改代码调试。  
2. 任意提名都能走完：提名 → 辩护结束 → 投票 → 结算 → 继续白天/进入夜晚。  
3. 恶魔被处决后，必定产生以下之一：  
   - `game.ended(winner=good)`  
   - `demon.changed(reason=scarletwoman)`（猩红女郎接管条件满足时）  
4. 自动超时推进开启后，AFK 不会导致流程永久卡死。

---

## 执行顺序（防返工）
先做“后端单一路径一致性 + 前端可操作性”再做超时自动化，避免在错误基础上叠功能。  
顺序：Phase 0 → Phase 1 → Phase 2 → Phase 3 → 验证。
