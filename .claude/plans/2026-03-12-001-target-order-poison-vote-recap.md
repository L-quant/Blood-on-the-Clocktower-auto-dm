# 2026-03-12 目标选择排序、中毒提示、投票播报与结算复盘修复

## 目标
- 修复选目标时座位号码乱序的问题，确保目标列表按 1 号到 N 号稳定排序。
- 修复角色中毒后出现不合理或误导性错误信息的问题，重点覆盖洗衣妇被中毒时的信息生成逻辑。
- 调整投票结果播报位置与内容，让投票结果进入【聊天】而不是【事件】，并替换原有【事件】里的投票播报。
- 修复游戏结束后未生成复盘的问题，确保结算后能稳定产出复盘内容。

## Bug 详情

### 1. 目标选择号码乱序
- 当前现象：玩家在需要选择目标时，目标列表中的座位号顺序不是按 1 号到 7 号递增展示，而是出现乱序。
- 影响范围：夜晚技能选人、白天技能选人、提名或其他复用目标选择器的场景都可能受影响。
- 预期结果：所有面向玩家的目标选择列表都应按 seat number 升序稳定展示；若存在不可选目标，也不能打乱其余目标的顺序。
- 验收口径：在 7 人房中，多次进入同一类目标选择面板，展示顺序始终为 1, 2, 3, 4, 5, 6, 7；若过滤掉自己或死亡玩家，剩余目标仍保持升序。

### 2. 中毒后出现不合理错误信息
- 当前现象：角色中毒后，系统会给出不合理、带有规则误导性的信息；已知例子是“洗衣妇中毒得知另外两个角色中有洗衣妇”这类明显不成立的提示。
- 影响范围：首夜信息角色、依赖“错误但看似合理信息”的角色提示生成逻辑，尤其是洗衣妇。
- 预期结果：中毒/醉酒时给出的错误信息必须满足“对玩家来说像是真的，但不应直接违反基础角色陈述结构或出现荒谬自指”。
- 验收口径：洗衣妇中毒时，不应出现“两名候选角色中有洗衣妇”这种把角色名本身当成候选结果的提示；错误信息应仍然是“两名玩家中有人是某个外来角色/镇民角色”这类结构合理的陈述。

### 3. 投票结果播报位置错误
- 当前现象：类似“1号 提 1号，投票玩家：2号、3号、4号、5号、6号、7号、1号 (7票)”的投票结果显示在【事件】中。
- 影响范围：白天提名/投票结算后的播报展示区分层。
- 预期结果：这类投票播报应出现在【聊天】中，用作主持播报；【事件】区不再保留原本那条重复或位置错误的投票播报。
- 验收口径：完成一次提名投票后，聊天区出现完整投票播报；事件区不再出现旧版重复投票结果文案，或已被新的正确事件表达替换。

### 4. 游戏结束后没有复盘生成
- 当前现象：游戏结束后没有自动生成复盘内容，导致结算后缺少总结输出。
- 影响范围：胜负结算后的 Auto-DM / summarizer / 异步任务链路。
- 预期结果：游戏进入结束态后，应触发一次复盘生成流程，并将结果保存或推送到前端可见位置。
- 验收口径：完成一局游戏并触发结束后，可以观察到复盘任务被触发且最终产出可见复盘内容；失败时至少应有明确日志，而不是静默缺失。

## 排查范围
- 前端：目标选择器、聊天区/事件区事件分发与展示逻辑、游戏结束界面或复盘展示入口。
- 后端：座位排序数据来源、夜晚信息生成与中毒扰动逻辑、投票结算事件类型与广播链路、游戏结束后的 Auto-DM/summary 触发流程。
- 异步依赖：RabbitMQ 任务投递、Auto-DM orchestrator、summarizer 子代理、前端消费复盘结果的状态同步链路。

## 验收标准
- 目标选择列表顺序稳定且符合座位号升序。
- 中毒后的错误信息“错误但合理”，不再出现洗衣妇示例中的荒谬提示。
- 投票结果从事件区迁移到聊天区，事件区旧播报被替换或移除。
- 游戏结束后能生成并看到复盘，或至少能在日志中清楚定位失败原因。
